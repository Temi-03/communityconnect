# ADR-002: Firebase over MongoDB with Express

## Date
31-10-2025

## Decision Owner
Tawakalitu Fetuga

---

## Context
Community Connect requires a backend solution to support user authentication, task creation, task acceptance, and task status updates. The application benefits from near real-time data updates, particularly when tasks are created or accepted by users.

The project community connect is developed within academic time constraints. As a result, ease of setup, reduced backend complexity, and maintainability are key considerations. Two backend approaches were evaluated:
- A managed backend using Firebase (Authentication and Cloud Firestore)
- A custom backend using Express.js with MongoDB

Industry documentation and architectural explanations were reviewed to understand the trade-offs between managed backend services and custom server-based solutions.

---

## Decision
We decided to use **Firebase (Authentication and Cloud Firestore)** instead of a **custom Express.js and MongoDB backend** for the Community Connect application.

---

## Decision Reasoning
The decision was driven by the following factors:
- Built-in user authentication without the need to implement custom authentication logic
- Real-time data synchronisation using Firestore listeners
- Reduced backend complexity, suitable for a solo developer
- Faster development and iteration within academic deadlines
- Automatic scaling and managed infrastructure with minimal configuration
- Reduced server maintenance and operational overhead

---

## Options Considered

### Option 1: Firebase (Authentication + Cloud Firestore)
- Provides managed authentication services
- Offers real-time NoSQL database capabilities
- Handles scaling and infrastructure automatically
- Integrates directly with client-side frameworks

### Option 2: MongoDB with Express.js
- Requires custom API development using Express.js
- Authentication and security must be implemented manually
- Greater flexibility in database querying and architecture
- Requires deployment and ongoing server maintenance

---

## Pros and Cons

### Firebase
**Pros**
- Built-in authentication with multiple sign-in methods
- Real-time updates using Firestore snapshot listeners
- Rapid development with minimal backend code
- Automatic scaling and managed infrastructure
- Well suited to solo development

**Cons**
- Vendor lock-in to Firebase services
- Query limitations compared to custom databases
- Potential cost increases at scale
- Requires careful configuration of security rules

### MongoDB with Express.js
**Pros**
- Full control over backend architecture and APIs
- Flexible querying and data modelling
- Easier migration between hosting providers
- No dependency on a single vendor ecosystem

**Cons**
- Increased development time and complexity
- Authentication and security must be implemented manually
- Ongoing server maintenance responsibility
- Additional effort required for scaling and deployment

---

## Consequences
Using Firebase enables faster backend development and simplifies authentication and real-time data handling, allowing the project to focus on core application features rather than infrastructure.  
While this introduces vendor lock-in and constraints on querying, these trade-offs are acceptable given the project scope and are mitigated through careful data modelling and security rule design.
